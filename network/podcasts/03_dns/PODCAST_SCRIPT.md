# 🎙️ Podcast Script: DNS (03_dns)

## 🎯 Introduction

오늘의 복습 주제는 **'DNS(Domain Name System)'**입니다. 숫자로 된 IP 주소 대신 우리가 기억하기 쉬운 도메인 이름을 이용해 길을 찾아주는 '인터넷의 전화번호부' DNS의 공학적 설계를 파헤쳐 보겠습니다.

---

## 🏛️ Section 1: 왜 분산 계층 구조인가? (The Decentralized Architecture)

1. **중앙 집중형의 한계**:
   - 만약 단 하나의 서버가 전 세계의 도메인을 관리한다면 어떨까요? 그 서버가 죽으면 인터넷이 멈추고(SPOF), 트래픽을 감당하지 못해 폭발할 겁니다. 물리적 거리가 멀면 응답 속도도 엄청나게 느려지겠죠.
2. **계층적 해법 (The Hierarchy)**:
   - 그래서 DNS는 **Root -> TLD(.com, .kr) -> Authoritative DNS(google.com)**로 이어지는 거대한 계층 구조로 설계되었습니다. 각 레벨에서 권한을 위임받아 관리하기 때문에 확장성과 안정성이 매우 높습니다.

## 🚀 Section 2: 질의 방식과 캐싱 (How it Works)

1. **Local DNS (대리인의 등장)**:
   - 유저가 직접 전 세계를 돌며 물어보지 않습니다. 우리 집 근처 ISP에 상주하는 **Local DNS**가 대신 발품을 팔아 최종 IP를 알아다 줍니다.
2. **반복적 질의 (Iterative Query)**:
   - Root 서버는 직접 답을 찾아주지 않습니다. "난 모르지만 .com 서버 주소는 알아, 저기 가서 물어봐!"라고 다음 갈 곳만 알려줍니다. 이것이 상위 서버의 부하를 줄이는 공학적 지혜입니다.
3. **캐싱(Caching)**:
   - 한 번 알아낸 주소는 일정 시간(TTL) 동안 메모리에 저장합니다. 가까운 곳에서 해결할 수 있게 함으로써 네트워크 지연을 극복합니다.

## 📜 Section 3: 레코드의 철학 (DNS Records)

1. **Type A (Address)**: 호스트이름과 IP 주소를 연결하는 실제 좌표입니다.
2. **Type NS (Name Server)**: "어떤 서버가 이 도메인의 진짜 권위자(Authority)인가?"를 증명하는 **임명장**입니다.
3. **Glue Record의 묘미**:
   - 네임서버의 IP를 알기 위해 네임서버에 물어봐야 하는 무한 루프를 해결하기 위해, TLD 서버가 NS 레코드와 A 레코드를 **풀(Glue)**로 붙여서 한꺼번에 던져주는 기막힌 수법입니다.

## 🎖️ Section 4: 시니어의 통찰 (Senior Insight)

- **정찰병 비유**:
  - DNS는 본격적인 본대(데이터)가 진격하기 전에 길을 닦는 **정찰병**입니다.
- **공학적 가치**:
  - DNS는 단순히 이름을 바꾸는 도구가 아니라, **SPOF 해결, 부하 분산, 속도 최적화, 확장성**이라는 4대 고학적 가치를 모두 담고 있는 분산 데이터베이스의 정수입니다.

---

## 🏆 Summary

DNS는 **'권한의 위임'**과 **'부하의 분산'**을 통해 수십억 개의 엔티티를 관리하는 장엄한 설계물입니다.
